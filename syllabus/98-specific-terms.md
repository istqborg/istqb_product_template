# Appendix D – Domain Specific Keywords

| Term Name | Definition |
| :-- | --- |
| Accessibility API | Platform interfaces provided by operating systems act as a bridge between applications and assistive technologies by exposing a programmatic representation of the user interface. |
| Accessibility Conformance Report (ACR) / VPAT | A formal document (often created using the VPAT® template) used for procurement and legal compliance that summarizes a digital product's level of conformance with specific accessibility criteria. |
| Accessibility Tree | A specialized data structure generated by the browser or operating system from the underlying UI structure that contains semantic information (roles, states, relationships) relevant to assistive technologies. |
| ARIA / WAI-ARIA | A technical specification published by the W3C that defines how to make dynamic web content and advanced user interface controls more accessible to people with disabilities through semantic markup attributes |
| Artificial intelligence | A field of computer science dedicated to creating systems capable of performing tasks that typically require human intelligence. In digital accessibility testing, AI is utilized to evaluate page context (e.g., suggesting alternative text) or to generate code corrections for accessibility violations. |
| Assistive technology (AT) | Any piece of equipment, software program, or product system that is used to increase, maintain, or improve the functional capabilities of persons with disabilities. |
| CI/CD pipeline | A set of automated software delivery processes. In the context of digital accessibility, these pipelines often incorporate automated quality gates to prevent the merging or deployment of code that contains critical accessibility defects. |
| Document Object Model (DOM) | A programming interface for web documents that represents the page structure. In accessibility testing, the fully rendered DOM is evaluated by UI-based automated scanners to detect dynamic accessibility defects that static code analysis might miss. |
| Linter | A static analysis tool used to analyze source code. In digital accessibility, linters are integrated into the developer's environment to provide real-time feedback and prevent structural accessibility defects during the active implementation phase. |
| Medical model (of disability) | A paradigm that views disability as an issue that exists within the person, caused by injury, illness, or a health condition, with the aim of diagnosing, treating, managing, or “fixing” the individual |
| POUR | An acronym representing the four foundational principles of the Web Content Accessibility Guidelines (WCAG), which state that digital content must be Perceivable, Operable, Understandable, and Robust. |
| Remediation | The process of correcting or fixing accessibility defects (barriers) within a digital product's code or design to bring it into compliance with established accessibility standards. |
| Screen magnifier | A type of assistive technology that enlarges on-screen content and enhances visual contrast, crucial for validating the accessibility experience for users with low vision. |
| Screen reader | Assistive technology that allows users to operate software without needing to view the visual display by converting digital text, interface elements, and structural code into synthesized speech or output on a refreshable braille display. |
| Semantic markup | The use of HTML markup to reinforce the meaning and structure of the information in web pages, rather than merely to define its visual presentation, facilitates native compatibility with assistive technologies. |
| Social model (of disability) | A paradigm that shifts focus away from the impairment and places responsibility on society, arguing that people are disabled by the physical, attitudinal, and systemic barriers that prevent them from participating equally. |
| Speech recognition | Assistive technology that enables interface navigation and text entry through spoken voice commands, used primarily by individuals with motor or mobility impairments. |
| Success criterion | A testable statement within the Web Content Accessibility Guidelines (WCAG) that defines a specific requirement for digital accessibility conformance. |
| Switch access | A type of assistive technology that allows users with significant motor impairments to navigate and interact with digital interfaces using one or more physical switches instead of a traditional mouse or keyboard. |
| Universal design | A design philosophy focused on creating products, environments, and systems that are usable by all people, to the greatest extent possible, without the need for adaptation or specialized design. |
| Voice control | A method of interacting with a digital interface using spoken commands, which requires testers to verify that visual elements possess predictable and programmatically accessible control names |
| WCAG2ICT | A guidance document published by the W3C that explains how to apply the Web Content Accessibility Guidelines (WCAG) principles, guidelines, and success criteria to non-web Information and Communications Technologies (ICT), such as native desktop software and mobile applications. |
| Web Content Accessibility Guidelines (WCAG) | A set of guidelines published by the Web Accessibility Initiative (WAI) of the W3C for making content accessible, primarily for people with disabilities. |
